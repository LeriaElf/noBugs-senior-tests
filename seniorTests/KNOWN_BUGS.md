# Известные баги и проблемы API

## 1. Разная структура ошибок на разных эндпоинтах

Сервер возвращает ошибки в разных форматах.

### Наблюдаемые форматы:

| Эндпоинт                  | Формат ошибки                                         | Пример                                                                                         |
| ------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `POST /admin/users`       | JSON объект, ключ — имя поля, значение — массив строк | `{ "username": ["Username cannot be blank", "Username must be between 3 and 15 characters"] }` |
| `POST /accounts/deposit`  | JSON объект, ключ — имя поля, значение — массив строк | `{ "amount": ["Deposit amount must be at least 0.01"] }`                                       |
| `POST /accounts/deposit`  | JSON объект с ключом "error" (строка)                 | `{ "error": "Unauthorized access to account" }`                                                |
| `POST /accounts/transfer` | JSON объект, ключ — имя поля, значение — массив строк | `{ "amount": ["Transfer amount cannot exceed 10000"] }`                                        |
| `POST /accounts/transfer` | JSON объект с ключом "error" (строка)                 | `{ "error": "Invalid transfer: insufficient funds or invalid accounts" }`                      |
| `PUT /customer/profile`   | Простая строка                                        | `"Name must contain two words with letters only"`                                              |

### Влияние:

- `ErrorHandlingRequester` должен обрабатывать и строки, и объекты, и массивы
- Тесты требуют разные значения `errorKey` в зависимости от эндпоинта и типа ошибки
- Нет единого контракта ошибок в API
- Валидационные ошибки возвращают массив сообщений по ключу-имени поля, бизнес-ошибки — строку по ключу `"error"`, а `/customer/profile` — просто строку без обёртки в объект

### Рекомендация:

Стандартизировать все ответы с ошибками в единый формат.

## 2. Swagger возвращает 400 для пустого balance, тесты получают 500

### Описание:

В Swagger UI при отправке запроса на депозит с пустым полем `balance` сервер возвращает `400 Bad Request`. Однако в автотестах аналогичный сценарий возвращает `500 Internal Server Error`.

### Payload в Swagger (из инспектора браузера):

```json
{"id": 28, "balance":}
```

### Тестовые payload и результаты:

| Payload                          | Ответ сервера                                      |
| -------------------------------- | -------------------------------------------------- |
| `{ id: 28, balance: null }`      | 500 Internal Server Error                          |
| `{ id: 28, balance: undefined }` | 500 Internal Server Error (поле опускается в JSON) |
| `{ id: 28, balance: "" }`        | 500 Internal Server Error                          |
| `{ id: 28, balance: "  " }`      | 500 Internal Server Error                          |

### Текущее состояние в тестах:

Тест `depositTests.js` ожидает `500 Internal Server Error` с комментарием `// API bug: returns 500 instead of 400`.

### Рекомендация:

- Сервер должен возвращать `400 Bad Request` для всех невалидных входных данных, включая null/undefined/пустые значения

## 3. Неопределённый формат суммы для транзакций (transfer/deposit)

### Описание:

Методы `generateTranferData` и `generateBalanceData` генерируют суммы в случайном формате — как целые числа, так и десятичные дроби (до 2 знаков после запятой).

### Текущее поведение:

Сервер принимает оба варианта чисел (целые и с десятичными дробями).

### Что требуется уточнить:

Какой формат суммы является допустимым согласно требованиям:

- Только целые числа
- Только числа с фиксированным количеством знаков после запятой (например, 2)
- Оба варианта

### Затронутые файлы:

- `seniorTest/models/accountTransferRequest.js` — метод `generateTranferData`
- `seniorTest/models/accountDepositRequest.js` — метод `generateBalanceData`

### Риск:

Пока формат не уточнён, тесты могут не покрывать граничные случаи или ошибочно проходить для невалидных данных.

## 4. Баланс возвращается с избыточной точностью (floating point precision bug)

### Описание:

API возвращает баланс с более чем 2 знаками после запятой из-за ошибок точности чисел с плавающей точкой.

### Наблюдаемое поведение:

| Операция | Ожидаемый баланс | Фактический баланс   |
| -------- | ---------------- | -------------------- |
| Трансфер | `3791.41`        | `3791.4100000000003` |
| Трансфер | `20650.49`       | `20650.489999999998` |
| Депозит  | `2648.22`        | `2648.2199999999993` |

### Текущее состояние в тестах:

В `transferBetweenAccaunts.js` добавлены проверки `senderDecimalPart.length <= 2` и `receiverDecimalPart.length <= 2`, которые упадут при появлении лишних знаков после запятой.

### Влияние:

- Невозможно корректно сравнивать балансы в тестах через `===` или `.to.equal()`
- Отображение баланса пользователю может содержать "мусорные" цифры
- Потенциальные проблемы при финансовых расчётах

## 5. Отсутствуют требования к полю name в документации API (PUT /customer/profile)

### Описание:

В документации API отсутствует информация о требованиях к полю `name` для эндпоинта `PUT /customer/profile`.

### Что не указано:

- Минимальная и максимальная длина имени (и каждого слова)
- Формат имени (например, "Имя Фамилия")
- Обязательность поля

### Расхождение модели и тестов:

### Предполагаемые правила:

- regex `/^[A-Za-z]{3,15}\s[A-Za-z]{3,15}$/`
- Формат: два слова, разделённые одним пробелом
- Каждое слово — только латинские буквы (A-Za-z)
- Минимальная длина 3 символа
- Максимальная длина 16 символов
- Не допускаются: цифры, спецсимволы, диакритические знаки (Élise), несколько пробелов, табуляция, перенос строки, пробел в начале

### Единое сообщение об ошибке:

Все невалидные имена возвращают одно сообщение: `"Name must contain two words with letters only"`. Нет отдельных сообщений для разных типов нарушений (длина, формат, спецсимволы), в отличие от валидации username/password, где каждая проблема описана отдельно.

### Влияние:

- Невозможно корректно валидировать данные на стороне клиента
- Тесты используют предполагаемые требования, которые могут не соответствовать реальной логике сервера
- Модель `PutCustomerProfileRequest` генерирует данные по regex, который может не совпадать с реальными границами сервера

### Рекомендация:

Добавить в документацию API описание требований к полю `name`.

## 6. Нестабильная ошибка при трансфере суммы >10000

### Описание:

В тесте на невалидную сумму трансфера `10000.99` закомментирована альтернативная ожидаемая ошибка с пометкой `// плавающий`:

```javascript
{
  amount: 10000.99,
  errorMessages: ["Transfer amount cannot exceed 10000"],
  // errorMessages: ["Invalid transfer: insufficient funds or invalid accounts"],// плавающий
}
```

### Влияние:

Сервер иногда возвращает `"Transfer amount cannot exceed 10000"` (валидационная ошибка), а иногда — `"Invalid transfer: insufficient funds or invalid accounts"` (бизнес-ошибка). Вероятно, это зависит от порядка проверок на сервере или от состояния баланса аккаунта-отправителя.

### Риск:

Тест может случайно падать при изменении условий (например, если баланс отправителя окажется меньше суммы трансфера, сервер может вернуть бизнес-ошибку вместо валидационной).
